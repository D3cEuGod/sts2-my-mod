#### 当前准备验证的伤害 / 负面效果分类

- 过量伤害（Overkill）
- 宠物 / 召唤物伤害
- 易伤增伤贡献（Vulnerable bonus damage）
- 虚弱减伤损失（Weak damage loss）
- 反应式追加伤害（reactive follow-up damage）
- 荆棘（Thorns）

#### 当前建议的默认统计口径

**默认计入主输出：**
- 玩家本人直接造成的普通伤害
- 归属到玩家的宠物 / 召唤物伤害
- 可明确归属到玩家的主动型追加伤害
- 易伤影响后的最终有效伤害

**默认单独记账，先不要直接混进主输出：**
- 过量伤害（建议单列）
- 易伤带来的额外增伤（建议单列贡献值）
- 虚弱导致的伤害损失（建议单列损失值）
- 荆棘 / 反伤
- HP loss / Osty 相关伤害
- 自伤 / 代价伤害
- 来源归属不清晰的反应式追加伤害

**验证版日志应覆盖：**
- 最终伤害结算（final damage result）
- 易伤倍率修改
- 虚弱倍率修改
- 荆棘 / 反伤相关触发
- 宠物 / 召唤物 owner 归属
- 过量伤害与格挡拆分

## 当前激活问题清单（2026-04-22）

### P0，先修这些，不然数据口径不稳

- [ ] **Run 边界错误，全局伤害跨局串数据**
  - 当前“全局游戏伤害”在一局结束后没有重新开始记
  - 新一局会把上一局的累计继续并入显示
  - 这会直接污染 run 级统计，是当前最高优先级

- [x] **补齐中毒 / 灾厄等 debuff 触发伤害的当前问题**
  - poison 已通过 combat-local source cache、first-tick cache seeding、以及 null-dealer history owner recovery 接回面板
  - doom 已通过 `DoomPower.DoomKill(...)` 的 defensive applier lookup 与 fallback attribution 接回面板
  - 后续若再扩新的 debuff 类型，继续沿用 isolated / additive 路径，不回到 broad HP-loss hooks

### P1，主链已基本可用后继续收口

- [x] **弱化 / 易伤调试日志降噪**
  - `WEAK_MOD` / `VULN_MOD` 仍保留为验证探针
  - 现已改成短时间窗口去重，避免同一计算在一场战斗里高频刷屏
  - 后续如要继续收口，应优先考虑按调试开关控制，而不是重新放大默认日志量

- [ ] **非攻击型伤害分类继续补齐**
  - 当前已看到荆棘相关触发，但还没有完整分类策略
  - 需要继续盘点：反伤、毒、中毒、灾厄、HP loss、Osty、自伤之外的持续型来源
  - 目标是区分哪些应该计入主输出，哪些应该单列
