# backlog

## 从 `examples/sts2/DamageMeter/` 值得借鉴的精简 backlog

> 这个列表只记录值得借鉴的产品方向，不把 binary strings 当成 API 证据。

### P1, 当前原型之后最值得做

- [ ] **本场 / 总局双视图**
  - 本场看单次战斗结算
  - 总局看当前 run 的累计表现
- [ ] **更清晰的显示模式**
  - 紧凑模式
  - 详细模式
  - 避免默认信息量过大
- [ ] **更多稳定的基础指标**
  - 总伤害
  - DPS / DPT
  - 最高单次命中
  - 过量伤害
- [ ] **更明确的结算边界**
  - 本场结束时结算
  - 新战斗开始时清本场
  - 总局保留到手动重置或新 run

### P2, 有了稳定数据后再考虑

- [ ] **分段视图**
  - Current / Fight / Overall 这种分层展示
- [ ] **更多分类页**
  - 输出
  - 防御
  - 资源
  - 分析
- [ ] **更强的明细 drill-down**
  - 按卡牌来源
  - 按伤害来源
  - 按玩家/宠物归属拆分
- [ ] **记录类统计**
  - 单回合最高伤害
  - 单场最高伤害
  - 最多出牌
  - 最高格挡

### P3, 明确不应现在就做

- [ ] 历史上传 / 排行榜 / 外部同步
- [ ] 多窗口 dashboard
- [ ] 大量 analytics / consent 流程
- [ ] 在没有稳定 API 证据前照搬 DamageMeter 的分类实现

## 当前建议顺序

1. 先把 `本场 / 总局` 做稳
2. 再确认伤害口径
3. 再做 compact / detailed 两种展示模式
4. 最后才考虑更多分类和 drill-down
